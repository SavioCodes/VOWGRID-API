# Auth Setup

## Scope

VowGrid now has real dashboard auth for human operators while keeping API keys for machine clients.

Implemented:

- Signup
- Login
- Logout
- Current session (`/v1/auth/me`)
- Password reset request and confirmation
- Email verification request and confirmation
- Workspace invite acceptance
- Workspace switching across active memberships
- Protected `/app` routes in the web app
- Session-backed dashboard data fetching
- Direct member, invite, and API key management for owners and admins
- Optional GitHub and Google OAuth login/signup

Partially implemented:

- Social login only becomes active when OAuth provider envs are configured

Not implemented:

- Enterprise SSO / SAML / generic OIDC
- Passwordless auth
- MFA

## Backend Model

Dashboard auth uses:

- `User.passwordHash`
- `User.emailVerifiedAt`
- `User.lastLoginAt`
- `User.disabledAt`
- `WorkspaceMembership`
- `WorkspaceInvite`
- `UserSession`
- `EmailVerificationToken`
- `PasswordResetToken`
- `OAuthAccount`
- `OAuthSignupToken`

Session tokens are:

- generated by the API
- hashed before persistence
- sent back to the web app on signup/login/invite acceptance/OAuth completion
- stored by the web app in an HttpOnly cookie

## Web Flow

1. A user signs up, logs in, completes an invite, or completes an OAuth callback.
2. A Next.js server action or route handler calls the matching API auth endpoint.
3. The web app stores the returned session token in `vowgrid_dashboard_session`.
4. Protected server components read that cookie and call the API with `Authorization: Bearer <token>`.
5. If the session is missing or invalid, `/app` redirects to `/login`.
6. Password reset and verification links return through `/reset-password` and `/verify-email`.
7. Accepted invites and additional memberships show up in the workspace switcher.

## Security Notes

- Passwords are hashed before storage
- Session tokens are hashed before persistence
- Password reset and email verification tokens are stored as hashes and expire automatically
- Disabled members have their active dashboard sessions revoked
- The browser only receives the opaque session token through an HttpOnly cookie
- API keys remain separate from dashboard auth and can be created, rotated, and revoked only by workspace owners or admins
